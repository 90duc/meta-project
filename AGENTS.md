# meta-project

当前项目是一个智能体应用，使用 **Java** 语言实现，包含 meta-ai 和 meta-agent 两个核心模块。
其中meta-ai作为meta-agent的大脑，负责解析用户的意图，转成工具的调用指令
meta-agent作为整个用户的入口，负责接受用户的输入，然后将用户的输入发送给meta-ai，从meta-ai获取工具的调用指令，按照步骤执行工具。
这是个动态智能交互循环的流程，不是一步解析就可以解析到位。

# meta-ai

meta-ai 是整个项目的大脑，负责解析用户输入，把用户输入转化成无歧义的指令（比如：某种编程语言）。

不借助 AI 大模型，要实现自然语言对指令的转换。

从大语言模型的原理分析出 AI 能力，是根据概率来生产内容的。这个模块也是根据概率来生成指令，帮忙设计方案。

## 核心能力

- **指令拆分**：将复杂输入拆分为多个独立或有序的指令
- **意图识别**：理解用户真实意图，消除歧义
- **方案生成**：基于概率模型生成可能的解决方案
- **指令编排**：确定多指令之间的执行顺序和依赖关系

## 实现逻辑

### 1. 解析循环（Parse Loop）

解析循环是 meta-ai 的核心机制，将自然语言转化为结构化指令。

```
用户输入 → 预处理 → 解析 → 评估 → 循环直到完成
```

#### 核心循环流程

```
┌─────────────────────────────────────────┐
│              Parse Loop                   │
│                                          │
│  ┌──────────┐                           │
│  │  Input   │◄──── 用户自然语言           │
│  └────┬─────┘                           │
│       ▼                                  │
│  ┌──────────┐                           │
│  │  Parser  │◄──── 模式库 + 规则库        │
│  └────┬─────┘                           │
│       ▼                                  │
│  ┌────────────────┐                     │
│  │ confidence?     │                    │
│     │       │                          │
│   Done    ┌──────────┐                  │
│           │  歧义处理  │                  │
│                │                         │
│                │ 返回澄清问题             │
│                └──────► 循环 ◄──────────│
└─────────────────────────────────────────┘
```

#### 循环终止条件

| 状态 | 动作 | 含义 |
|------|------|------|
| `resolved` | 停止循环 | 置信度达标，输出结构化指令 |
| `ambiguous` | 生成问题，继续循环 | 需要用户澄清意图 |

### 2. 文本预处理

```
用户输入 → 规范化 → 分词 → 词性标注 → 依存分析 → 结构化表示
```

#### 2.1 规范化
- **大小写统一**：转换为小写或保持原样
- **标点处理**：去除多余标点，统一分隔符
- **编码标准化**：Unicode 规范化

#### 2.2 分词

采用离线 SDK 实现，参考大模型的分词策略：

- **中英文混合处理**：
  - 中文：基于词典的逆向最大匹配（RMM）+ N-gram 新词发现
  - 英文：空格分词 + 子词分割（BPE）

- **Java 离线分词工具**：
  - **HanLP**：功能完善，性能高效，支持自定义词典
  - **IKAnalyzer**：轻量级中文分词，支持细粒度和智能切分
  - **paoding**：基于字典的中文分词，支持不限制个数的词组切分

- **分词策略**：
  - 优先使用词典匹配
  - 未登录词采用字符级分词
  - 中英文混合文本统一处理
  - 特殊符号独立保留

#### 2.4 依存分析

基于 L. Tesnière 的依存语法理论，分析词与词之间的支配与被支配关系。

##### 依存关系类型

| 关系类型 | 标签 | 说明 |
|----------|------|------|
| 主谓关系 | SBV | subject-verb |
| 动宾关系 | VOB | verb-object |
| 间宾关系 | IOB | indirect-object |
| 定中关系 | ATT | attribute |
| 状中关系 | ADV | adverbial |
| 补语关系 | CMP | complement |
| 并列关系 | COO | coordinate |
| 介宾关系 | POB | preposition-object |
| 左附加关系 | LAD | left-adjunct |
| 右附加关系 | RAD | right-adjunct |
| 独立结构 | IS | independent structure |
| 核心关系 | HED | head |

##### 实现算法

**基于转移的分析（Transition-based）**：
- **Arc-standard**：标准弧分析，使用移进、归约操作
- **Arc-eager**：贪婪弧优先，减少栈操作
- **Arc-hybrid**：混合策略，平衡效率和准确率

**基于图的分析（Graph-based）**：
- 暴力搜索所有可能的生成树
- 准确率高但计算量大

##### Java 离线实现工具

#### 2.5 指令拆分

用户输入可能是多个连续指令，需要基于语义理解拆分为独立的指令序列。

```
用户输入 + 工具能力 → 语义映射 → 指令拆分 → 依赖关系 → 执行计划
```

##### 工具能力获取

meta-agent 调用 meta-ai 时，会将可用工具的能力描述一并传入：

```
meta-agent → meta-ai.parse(input, tools) → 结构化指令
```

```json
{
  "tools": [
    {
      "name": "weather.get",
      "capability": "查询天气数据，返回指定地点的温度、湿度、风力等信息",
      "input_type": "location, date",
      "output_type": "weather_data"
    },
    {
      "name": "email.send",
      "capability": "发送邮件到指定收件人，支持主题和内容",
      "input_type": "recipient, subject, content",
      "output_type": "email_result"
    }
  ]
}
```

##### 语义动作映射

基于工具能力描述动态识别语义动作：

```
用户输入 → 向量化 → 语义相似度匹配 → 工具候选排序 → 动作识别
```

**匹配流程**：
1. 将用户输入向量化（Word2Vec / 句子向量）
2. 将工具 capability 向量化
3. 计算用户输入与各工具 capability 的余弦相似度
4. 按相似度排序，筛选高于阈值的工具

**相似度计算**：
```
similarity = cosine(vector(user_input), vector(tool.capability))
```

**动作类型由工具决定**：
- 工具的 `capability` 决定动作类型
- 动作命名遵循：`{工具名}.{操作名}` 格式
- 例如：`weather.get`、`email.send`、`file.create`

##### 指令拆分算法

**基于依存分析的指令边界识别**：
- 识别句子中的核心动词作为指令标识
- 根据依存关系确定每个动作的主语和宾语
- 将完整的动宾结构提取为独立指令

**基于语义角色的参数提取**：
- Agent（执行者）：动作的执行主体
- Patient（受事者）：动作的承受对象
- Instrument（工具）：执行动作的手段
- Goal（目标）：动作指向的目标位置

**示例解析**：

输入：`查询今天的天气，发送到邮件上`

工具匹配：
```
weather.get (相似度: 0.92) ← 语义匹配 "查询天气"
email.send (相似度: 0.85) ← 语义匹配 "发送邮件"
```

语义分析结果：
```
├── 指令1：weather.get
│   └── 参数：location=当前位置, date=今天
└── 指令2：email.send
    └── 参数：recipient=用户邮箱, content=天气数据
```

##### 依赖关系推断

基于工具的输入输出类型推断指令间的数据依赖：

**数据流依赖**：
- 前一个工具的 `output_type` 作为后一个工具的输入
- 例如：`weather.get` 输出 → `email.send` 的 content 参数

**时序约束**：
- 存在数据依赖时，前置动作必须先完成
- 无数据依赖时，可并行执行

**类型匹配**：
- 输出类型与输入类型需匹配
- 例如：JSON → 文本（需转换器）

##### 执行计划生成

```
指令序列 + 依赖关系 → DAG 图 → 并行分组 → 执行计划
```

```json
{
  "instructions": [
    {
      "id": 1,
      "action": "weather.get",

### 3. 语义向量化

将文本转换为向量表示，实现语义级别的理解。

```
用户输入 → 向量化 → 语义空间映射 → 意图概率分布
```

#### 3.1 文本表示

- **词袋模型（BOW）**：词频-逆文档频率（TF-IDF）权重
- **词向量（Word2Vec）**：词语的分布式表示
- **句子向量**：对词向量加权平均或使用 SIF 算法
- **稠密向量**：通过矩阵分解将稀疏特征转为低维稠密向量

#### 3.2 语义空间
- **意图空间**：预定义的意图向量簇
- **语义距离**：计算输入向量与意图簇中心的余弦距离
- **概率分布**：Softmax 归一化得到各意图的后验概率

```
P(intent|input) = softmax(cosine(vector(input), intent_centers))
```

#### 3.3 TF-IDF 向量化

将文本转换为词频-逆文档频率向量：

- **词频（TF）**：统计词在文档中出现的次数
- **逆文档频率（IDF）**：log(文档总数/包含该词的文档数)
- **权重计算**：TF × IDF，突出重要词，抑制常见词

引用库：**Lucene**

#### 3.4 Word2Vec 词向量

将词语映射到低维连续向量空间：

- **Skip-gram**：根据中心词预测上下文
- **CBOW**：根据上下文预测中心词
- **训练过程**：神经网络反向传播，学习词间语义关系

引用库：**DL4J**、**W2V4J**、**HanLP**

#### 3.5 句子向量

将句子转换为固定维度的向量表示：

- **平均词向量**：对句子中所有词的向量取平均
- **SIF 算法**：加权平均 + PCA 主成分移除，消除常见词影响
- **加权公式**：a / (a + frequency(w))

引用库：**SIF-Java**

#### 3.6 语义相似度计算

- **余弦相似度**：cos(θ) = (A·B) / (||A|| × ||B||)
- **欧氏距离**：两点间的直线距离
- **语义匹配**：基于向量距离判断语义相近程度

引用库：**Apache Commons Math**、**EJML/MTJ**

### 4. 概率推理引擎

基于贝叶斯网络进行意图推理和方案生成。

```
先验概率 → 似然估计 → 后验推断 → 期望最大化
```

#### 4.1 贝叶斯网络结构

```
意图(I) ──→ 操作(O) ──→ 参数(P)
   │              │
   └──→ 上下文(C) ──→ 约束(R)
```

- **节点**：意图、操作、参数、上下文
- **边**：因果依赖关系
- **条件概率表（CPT）**：每个节点在父节点取不同值时的概率分布

#### 4.2 概率计算

**链式法则**：
```
P(intent, action, params | input) = P(params | action) * P(action | intent) * P(intent | input)
```

**贝叶斯推断**：
```
P(intent | observed) = α * P(observed | intent) * P(intent)
```

#### 4.3 概率传播
- **正向传播**：从输入推断意图，再推断操作和参数
- **反向传播**：从观察到的结果反推最可能的意图
- **消息传递**：信念传播算法（Belief Propagation）计算边缘概率

### 5. 方案生成（概率推理）

基于概率模型生成最可能的执行方案，支持单指令和多指令场景。

```
意图分布 → 操作候选 → 概率排序 → 方案输出
```

#### 5.1 操作候选生成
- **意图-操作映射**：一个意图可能对应多个操作
- **上下文约束**：根据当前上下文筛选可用操作
- **历史偏好**：用户历史行为影响操作排序
- **多指令扩展**：同一输入可能包含多个意图，生成指令序列

#### 5.2 概率排序

**综合得分**：
```
Score(action) = P(action | intent) * P(params | action, context) * Boost(history)
```

- **条件概率**：P(action | intent) 从规则库获取
- **参数匹配度**：操作所需参数的完整度和匹配度
- **历史提升**：Boost(history) 基于用户历史选择调整

#### 5.3 指令序列生成

当输入包含多个指令时，需要生成有序的指令序列：

- **顺序预测**：基于动词和时间连词确定执行顺序
- **依赖分析**：识别指令间的数据依赖关系
- **并行分组**：将无依赖的指令分为同一并行组
- **序列优化**：考虑指令间的数据传输和状态共享

```json
{
  "instruction_sequence": [
    {
      "id": 1,
      "action": "file.create",
      "params": {"path": "user.sql"},
      "parallel_with": []
    },
    {
      "id": 2,
      "action": "data.insert",
      "params": {"source": "user.sql"},
      "parallel_with": []
    }
  ],
  "parallel_groups": [[1], [2]]
}
```

#### 5.4 Top-N 输出
- **方案多样性**：确保候选方案覆盖不同路径
- **置信区间**：返回每个方案的概率置信区间
- **可解释性**：附带概率来源的解释

### 6. 歧义处理（概率消解）

当置信度不足时，通过概率方法消解歧义。

```
置信度不足 → 候选分析 → 差异提问 → 用户选择 → 概率更新
```

#### 6.1 歧义检测
- **最大概率阈值**：低于阈值触发消解
- **概率差距**：Top-2 方案概率接近时触发
- **参数缺失**：关键参数缺失时触发

#### 6.2 差异分析
- **候选对比**：分析 Top-N 方案的关键差异点
- **关键问题**：生成最小化问题集的澄清问题
- **选项生成**：提供二选一或三选一的选项

#### 6.3 概率更新
```
P'(intent) = P(intent) * Weight(user_selection) / Z
```
- 根据用户选择更新概率分布
- 历史数据用于后续学习的先验调整

### 7. 上下文感知

利用历史上下文提升预测准确度。

```
历史上下文 → 特征增强 → 概率调整 → 上下文更新
```

#### 7.1 上下文特征
- **近期意图**：最近 1-3 次对话的意图类型
- **操作序列**：用户的操作习惯和偏好
- **时间模式**：用户在不同时间的意图分布差异
- **会话主题**：当前会话的主题一致性

#### 7.2 马尔可夫模型
- **状态**：当前意图
- **转移概率**：P(intent_t | intent_{t-1}, intent_{t-2}, ...)
- **发射概率**：P(input | intent)

```
P(intent_t | history) = Σ P(intent_t | intent_{t-1}) * P(intent_{t-1} | history)
```

#### 7.3 上下文权重
- **衰减因子**：历史越久远，影响越小
- **衰减函数**：指数衰减或线性衰减

### 8. 学习与适应

从用户反馈中持续学习和优化概率模型。

```
用户反馈 → 反馈收集 → 参数更新 → 模型优化 → 性能评估
```

#### 8.1 增量学习
- **在线学习**：每次交互后更新概率参数
- **经验回放**：保留历史样本进行批量更新
- **早停策略**：防止过拟合

#### 8.2 参数更新

**最大似然估计（MLE）**：
```
θ_new = argmax_θ ∏ P(user_choice | θ, context)
```

**贝叶斯更新**：
```
P(θ | data) ∝ P(data | θ) * P(θ)
```

#### 8.3 模型评估
- **困惑度（Perplexity）**：模型对未见数据的预测能力
- **准确率**：Top-1 意图识别的正确率
- **覆盖率**：成功解析的输入比例

# meta-agent

meta-agent 是整个项目的入口，主要负责接收用户的输入，记录用户的上下文窗口，提供可以执行工具（第三方 API 接口、编程环境）等等。

参照 Claude Code 的工具来设计。

## 核心能力

- 用户输入处理：接收并解析用户命令
- 上下文管理：维护对话历史和上下文窗口
- 工具调用：集成第三方 API、编程环境等执行工具
- 结果反馈：将执行结果返回给用户

## 实现逻辑

### 1. Agent Loop（智能体循环）

Agent Loop 是 meta-agent 的核心机制，将用户请求转化为多步骤的自主执行流程。

```
用户输入 → 模型评估 → 工具调用 → 结果反馈 → 循环直到完成
```

#### 核心循环流程（TAOR）

```
┌─────────────────────────────────────────┐
│              Agent Loop                  │
│                                          │
│  ┌──────────┐                           │
│  │  Prompt   │◄──── 用户消息              │
│  └────┬─────┘                           │
│       ▼                                  │
│  ┌──────────┐                           │
│  │   模型    │◄──── 系统提示 + 历史消息   │
│  └────┬─────┘                           │
│       ▼                                  │
│  ┌────────────────┐                     │
│  │  stop_reason?   │                    │
│     │       │                          │
│   Done    ┌──────────┐                  │
│           │  工具调用  │                  │
│                │                         │
│                │ 返回结果到消息            │
│                └──────► 循环 ◄──────────│
└─────────────────────────────────────────┘
```

#### 循环终止条件

| stop_reason | 动作 | 含义 |
|-------------|------|------|
| `end_turn` | 停止循环 | 模型已完成任务，返回结果给用户 |
| `tool_use` | 执行工具，继续循环 | 模型需要调用工具 |

### 2. 四个执行阶段

在每次循环中，模型会经历以下四个阶段：

```
Gather → Plan → Act → Verify
  ↑                      │
  └──────────────────────┘
```

- **Gather（收集）**：使用 Read、Glob、Grep、Bash 了解当前状态
- **Plan（规划）**：使用 TodoWrite 创建任务清单，制定执行计划
- **Act（执行）**：使用 Edit、Bash 等工具执行具体操作
- **Verify（验证）**：检查执行结果，必要时循环修正

### 3. 消息管理

```
消息数组 = [
  { role: "user", content: "用户请求" },
  { role: "assistant", content: [工具调用块] },
  { role: "user", content: 工具执行结果 },
  ...
]
```

- 消息数组随循环累积增长
- 工具结果自动追加到消息历史
- 支持并行工具调用（单次可调用多个工具）

### 4. 上下文管理

```
新输入 → 上下文窗口检查 → 历史裁剪 → 新内容追加 → 上下文快照
```

- **窗口大小**：定义最大保留历史条目数（如 100 条）
- **重要性评估**：根据意图类型标记内容权重
- **历史裁剪**：超出窗口时，按权重和时效裁剪旧内容
- **快照机制**：定期保存上下文状态，支持恢复
- **自动压缩**：上下文使用约 90% 时自动触发压缩

### 5. 子智能体机制

对于复杂任务，启用子智能体进行探索：

```
主循环 → Task(子智能体) → 独立上下文 → 结果汇总 → 主循环继续
```

- 子智能体拥有独立的上下文空间
- 探索结果作为工具输出返回主循环
- 避免探索过程污染主会话上下文

### 6. 工具调用机制

```
任务对象 → 工具匹配 → 参数映射 → 工具执行 → 结果标准化 → 反馈生成
```

- **工具注册表**：维护可用工具列表及元信息
- **工具匹配**：根据意图类型选择对应工具
- **参数映射**：将任务参数转换为工具所需格式
- **执行与重试**：调用工具，失败时自动重试（N次）
- **结果标准化**：统一不同工具的返回格式
- **批量执行**：支持多个指令批量执行
- **并行调度**：根据指令依赖关系和并行分组进行并行执行

### 8. 工具权限管理

- **allow**：自动允许执行的操作
- **ask**：执行前需用户确认
- **deny**：禁止执行的操作

示例配置：
```json
{
  "permissions": {
    "allow": ["Bash(npm run *)", "Bash(git status)"],
    "ask": ["Bash(git push:*)", "Bash(npm install *)"],
    "deny": ["Read(./.env*)", "Read(./secrets/**)"]
  }
}
```

### 9. 钩子机制（Hooks）

在工具调用的关键节点注入自定义逻辑：

| 钩子点 | 时机 |
|--------|------|
| `UserPromptSubmit` | 用户提交输入后处理前 |
| `PreToolUse` | 工具执行前（可拦截/修改） |
| `PostToolUse` | 工具执行后 |
| `Pause` | 模型暂停时 |
| `Resume` | 模型恢复时 |

### 10. 结果反馈

```
执行结果 → 结果评估 → 格式化输出 → 流式/批量推送 → 用户展示
```

- **结果评估**：检查执行是否成功，生成状态码
- **格式化**：将结果转为人类可读格式（文本、表格、图表）
- **推送方式**：支持流式输出或批量返回
